Задача

Убрать ручные уточнения из процесса записи

Небольшие компании часто принимают записи в сообщениях. Сотруднику приходится несколько раз уточнять услугу, специалиста, дату и время, а записи могут потеряться или пересечься. Этот проект показывает, как собрать этот путь в одну связанную систему.

Решение

Компоненты одной системы

01

Telegram-бот

Ведёт клиента по шагам: от выбора услуги и сотрудника до подтверждения времени.

02

REST API

Принимает запросы бота и админ-панели, выдаёт свободные слоты и сохраняет записи.

03

PostgreSQL / Neon

Хранит записи, клиентов, услуги, сотрудников и настройки системы.

04

Админ-панель

Показывает бронирования, помогает менять их статусы и видеть актуальный список записей.

Пользовательский сценарий

От выбора услуги до записи в панели

  1. Клиент открывает Telegram-бота
  2. Выбирает услугу и сотрудника
  3. Выбирает дату и свободное время
  4. Подтверждает данные
  5. API сохраняет запись
  6. Администратор видит её в панели

API, данные и deployment

Разделённая, но связанная архитектура

Система разделена на три части, а REST API остаётся единым слоем бизнес-логики. PostgreSQL используется как источник данных, а доступ администратора и Telegram-бота защищён разными способами авторизации.

Express подготовлен для serverless-развёртывания на Vercel. Конфигурация подключений и секреты передаются через переменные окружения и не попадают в публичный код.

Challenges & lessons

Что потребовало внимания

  • Связать бота, API и админ-панель в один сценарий.
  • Настроить CORS и production-переменные окружения.
  • Подключить PostgreSQL / Neon и выдачу свободных слотов.
  • Не допустить повторную запись на одно время.
  • Сохранить корректную работу Express в serverless-среде.

Главный вывод из этой работы: интерфейс, бот и база данных должны быть связаны через понятный API-контракт, иначе развитие системы быстро становится сложнее.

Текущее ограничение

Состояние диалога бота пока хранится в памяти процесса

При serverless cold start текущий шаг пользователя может сброситься. В следующей версии это состояние можно перенести в PostgreSQL или Redis. Это известное ограничение текущей архитектуры, а не часть данных бронирования.
